# 大模型"黑话"指南 — 给非算法开发者的速通版

> 本文写给：
> - 每天调 API 但不知道里面在干什么的开发者
> - 面试被问到"你了解 Flash Attention 吗"时不想只说"啊我听说过"
> - 想看懂技术文章里的各种英文缩写

---

## 前言：一条原则

每个技术名词背后都对应一个**具体的问题**。

| 名词 | 要解决的问题 |
|:-----|:------------|
| Attention | 模型看输入时怎么知道"哪部分更重要" |
| Flash Attention | Attention 算得太慢了 |
| MOE | 模型太大会变慢 |
| KV Cache | 对话生成太慢 |
| GQA | KV Cache 吃太多内存 |
| RoPE | 模型不知道"第一句话"和"第十句话"的位置区别 |
| Speculative Decoding | 大模型一个字一个字生成太慢了 |
| LoRA | 微调整个模型成本太高 |

> 每个技术 = **遇到了什么问题 + 怎么解决的**。记住这个，黑话就不可怕了。

## 1. Attention：模型在看什么

### 一句话

> 模型在生成每个词时，**给自己的输入按重要性打分**，重要的词权重高，不重要的权重低。

### 举个例子

你问："**北京的** **天安门**在哪？"

模型在生成答案时，对"北京"的关注权重一定比对"的"大很多。这种"关注哪些词"的机制就是 Attention。

### 为什么叫 Attention？

直译过来就是"注意力"——模型把注意力集中在最重要的部分上。

$$ \text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V $$

**开发者视角：**
- Q = Query（你要查什么）
- K = Key（有哪些可查的内容）
- V = Value（查到的实际信息）
- 这不就是 **KV 数据库的查询**吗？

### 实操意义

| 你知道就行 | 你不知道也没事 |
|:-----------|:---------------|
| Attention 是 Transformer 的核心 | 具体的 softmax 公式怎么推导 |
| 上下文越长，计算量越大（O(n²)） | 为什么是除以 sqrt(d_k) |
| 这就是为什么长文本更贵 | |

## 2. MHA → GQA → MQA：Attention 的三个变体

### 先说为什么有这些变体

Attention 需要一个**多头**（Multi-Head）机制：

```
想象 8 个侦探同时看一份证据：
- 侦探1 重点关注"时间"
- 侦探2 重点关注"地点"
- 侦探3 重点关注"人物"
...
8 个侦探各看各的，最后汇总意见。
```

这 8 个"侦探"就是 8 个 **Head**。

### MHA（Multi-Head Attention）

原始方案。每个 Head 有自己独立的 Q、K、V。

```
Layer：    Q₁ K₁ V₁    Q₂ K₂ V₂    Q₃ K₃ V₃    ... Q₈ K₈ V₈
推理时：  每个 Head 的 K 和 V 都要缓存（KV Cache）
内存：    = 头数 × 每头的缓存大小
```

**问题：** 8 个头 = 8 份 K 和 V，内存消耗大。

### MQA（Multi-Query Attention）

> 用了才发现，大部分 Head 共享 K 和 V 也没啥区别。

```
改进：    Q₁      Q₂      Q₃      ... Q₈
          ↓       ↓       ↓           ↓
        同一个 K₀ ───── V₀ ─────────────
```

| | MHA | MQA |
|:-|:----|:----|
| KV 缓存 | 8 份 | 1 份 |
| 内存 | 高 | **降低 ~87%** |
| 质量 | 基准 | 轻微下降（几乎看不出） |

**代表模型：** PaLM、Gemma

### GQA（Grouped Query Attention）

MHA 和 MQA 的折中。把 Head 分成几组，组内共享 K 和 V。

```
Llama 2 70B 的参数：
- 64 个 Head
- 分成 8 组
- 每组 8 个 Head 共享 1 份 KV
- 缓存 = 8 份（而不是 64 份）
```

| | MHA | GQA | MQA |
|:-|:----|:----|:----|
| 缓存份数 | N | G（G < N） | 1 |
| 效果 | 基准 | 几乎无损 | 轻微下降 |
| 代表 | 原始 Transformer | **Llama 2/3** | PaLM |

**为什么主流模型都转 GQA：** 推理速度几乎翻倍，质量几乎不掉。

## 3. Flash Attention：让 Attention 变快

### 核心问题

不是数学问题，而是**硬件问题**。

```
GPU 架构：
┌──────────────────────────────────┐
│          SRAM（超级快，很小）       │  ~20MB  ← 想在这算
│            ↓（太慢了！）            │
│          HBM（大，但慢很多）        │  ~80GB  ← 数据在这存着
└──────────────────────────────────┘

传统 Attention 的做法：
1. 从 HBM 把整个矩阵 Q, K, V 读到 SRAM  ← 这一步巨慢
2. 在 SRAM 算 softmax
3. 把结果写回 HBM
→ 瓶颈在数据搬运，不在计算
```

### Flash Attention 的解法

> 不一次算完，**分块算**。

```
传统方式：一次读完 → 算 → 写回     就像去图书馆借百科全书整套
Flash：   分块读 → 算一块 → 合并   就像每次只借一册
```

**效果：**

| | 传统 Attention | Flash Attention |
|:-|:---------------|:----------------|
| 速度 | 基准 | **2-4x 更快** |
| 内存 | O(n²) | O(n) |
| 精度 | 基准 | 完全相同 |
| 最长序列 | ~8K tokens | **128K+ tokens** |

### 作为开发者

- 这不是模型架构的改变，是**工程实现**的优化
- 使用 Flash Attention 不需要改模型代码
- HuggingFace 的 `attn_implementation="flash_attention_2"` 一行生效
- **Flash Attention 2（2023）** 和 **Flash Attention 3（2024）** 依次更快

## 4. KV Cache：对话为什么越聊越慢

### 问题

你问 AI：
```
"北京有什么好玩的？"           → 很快
"天安门呢？"                   → 快
"那故宫呢？故宫和天安门远吗？"  → 明显变慢
```

### 原因

Transformer 生成时，每生成一个字，都要"回顾"前面所有的字。

```
生成第 1 个字：回顾  →  输出"北"
生成第 2 个字：回顾北京  →  输出"京"
生成第 3 个字：回顾北京有  →  输出"有"
...
```

每次回顾都要重新计算前面的 K 和 V，**重复劳动**。

### 解法

```
第1次生成：
  计算：Q₁, K₁, V₁  →  第一个词
  缓存：把 K₁, V₁ 存起来

第2次生成：
  计算：Q₂  ← 只算新的
  读取：K₁, V₁  ← 直接用缓存的
  输出：第二个词

第N次生成：
  计算：Qₙ  ← 永远只算新的
  读取：K₁...Kₙ₋₁, V₁...Vₙ₋₁  ← 从缓存读
  输出：第 N 个词
```

**效果：** 生成速度从 O(n²) 降到 O(n)。

### 问题

KV Cache 吃的是**显存**，不是内存。

| 模型 | 每个 Token 的 KV Cache |
|:-----|:-----------------------|
| Llama 2 7B | ~1MB |
| Llama 2 70B | ~10MB |
| 输出 2048 tokens | ~20GB（70B 模型） |

> 这就是为什么长对话经常 OOM（显存溢出）。

### PagedAttention（vLLM）

类比操作系统的**虚拟内存**：

```
传统 KV Cache：  整个对话的 K, V 我全占着，谁也别碰  ← 碎片化严重
PagedAttention： 把 KV Cache 分成 16 页，按需分配   ← 几乎无浪费
```

vLLM 靠这个技术把推理吞吐提升了 **2-4x**。

## 5. MOE（Mixture of Experts）：模型大但不慢

### 问题

要让模型变聪明，通常需要**变大**。但变大了就**变慢**。

### 常规思路 vs MOE

```
传统模型：
输入 → [ 一个巨无霸神经网络 ] → 输出
       所有参数全部激活
       175B = 175B 都干活

MOE 模型：
输入 → Router（路由） → 专家1    ← 只激活一部分
                       专家3    ← 比如 2/8
                       专家7
                     其他5个专家休息
```

### 类比

```
传统模型：
   一个大公司的全部员工（100人）同时给你服务
   效率低，成本高

MOE：
   一个 100 人的大公司，但只有 2 个最对口的专家给你服务
   剩下 98 人在休息
```

### 关键数字

| | 传统 Dense 模型 | MOE 模型 |
|:-|:---------------|:---------|
| 总参数量 | 175B | 1T（1万亿） |
| 每次激活 | 175B | **~40B** |
| 推理速度 | 基准 | **接近 2-4x 快** |
| 训练成本 | 高 | 更高 |

**代表模型：**
- Mixtral 8x7B（每次激活 ~12B，但效果像 46B 模型）
- DeepSeek V2/V3
- GPT-4（据传也是 MOE）

### 对开发者的影响

使用 MOE 模型时（比如 Mixtral），**不需要任何特殊代码**。API 调用方式和普通模型一样。

## 6. RoPE：让模型知道"顺序"

### 问题

Attention 本身没有顺序感。

```
"我打你" → Attention 看到的
"你打我" → Attention 看到的
在数学上居然是一样的！因为"我"和"你"只是交换了位置，词向量一样。
```

所以需要**位置编码**来告诉模型"哪个词在前，哪个在后"。

### RoPE（Rotary Position Embedding）

> 把位置信息**旋转**到词向量里。

```
想像一个二维坐标：
  
  位置 1 的词：  向量指向 0°
  位置 2 的词：  向量指向 45°
  位置 3 的词：  向量指向 90°
  ...
  位置 n 的词：  向量指向 n × θ°

两个词"离得越远"，它们的向量方向差越大。
```

**为什么叫 Rotary：** 每次新位置就是在原有向量上转了角度。

### RoPE 为什么流行

| 方案 | 问题 |
|:-----|:-----|
| 以前用的 Absolute Position Embedding | 训的时候最长 2048，上线不能超 |
| RoPE | **位置相对关系**，可以外推到更长 |

所以 Llama、Mistral、Qwen 都用 RoPE。

### 长度外推

这就是为什么做 **Rope Scaling**（比如 NTK-aware scaling）可以让模型支持更长的上下文：

```
原始 Llama 2：      训练 4K → 推理 4K
+ Rope Scaling：    训练 4K → 推理 32K（质量略降）
+ YaRN：            训练 4K → 推理 128K
```

## 7. Speculative Decoding：让生成变快

### 问题

大模型一个字一个字地生成：

```
你问："写个关于Python的笑话"
模型： P（思考 ~30ms）
模型： y（思考 ~30ms）  
模型： t（思考 ~30ms）
模型： h（思考 ~30ms）
模型： o（思考 ~30ms）
模型： n（思考 ~30ms）
...
200 个字 × 30ms = 6 秒
```

### 解法

> 用一个**更小的模型**当"助理"，先快速写一堆，大模型快速检查。

```
步骤1：小模型（比如 0.5B）快速猜 5 个字
       小模型： "Python调包"
       花了：  3ms

步骤2：大模型（比如 70B）批量检查这 5 个字
       大模型： "P-y-t-h-o" ✓ ✓ ✓ ✓ ✓ 全对，收下
       花了：  30ms（一次性检查 5 个）

进度：5 个字只用了 33ms，而不是 150ms
加速：约 4-5 倍
```

**关键：** 大模型检查 1 个字和检查 5 个字花的时间差不多。

### 实际效果

| 场景 | 加速比 |
|:-----|:------|
| 短文本 | 1.5-2x |
| 长文本 | 3-5x |
| 代码生成 | 2-3x |

**在用的产品：** 基本上所有大厂的后端推理都在用。

## 8. Quantization：给模型"减肥"

### 问题

模型太大，显存放不下。

```
Llama 2 70B：
参数：70B × 每个参数 2 字节（FP16）= 140GB
一张 A100 只有 80GB，放不下
```

### 解法

> 把"精确但不重要"的小数精度砍掉。

```
精度等级（从高到低）：

FP32（32位浮点）= 1200 元的电子秤，精确到 0.01g  ← 训练时用
FP16（16位浮点）= 50 元的厨房秤，精确到 1g     ← 推理时用
INT8（8位整数） = 手掌大小的弹簧秤             ← 推理时用，质量几乎不掉
INT4（4位整数） = 单手掂一下"大概半斤"          ← 质量可能下降
```

| 精度 | 大小（70B 模型） | 效果损失 |
|:-----|:-----------------|:---------|
| FP16 | 140GB | 0% |
| INT8 | 70GB | <1% |
| INT4 | 35GB | 2-5% |
| NF4（QLoRA） | 35GB | 接近 INT8 |

**关键：** 这不仅仅是压缩——量化后模型可以跑在消费级显卡上。

```
70B 模型：
FP16 → 需要 2 张 A100（每张 80GB，约 ¥10万/张）
INT4 → 1 张 RTX 4090（24GB，约 ¥1.5万）   ← 家里就能跑！
```

### 实操

- **GGUF**：Llama.cpp 用的量化格式
- **GPTQ**：另一种量化格式
- **AWQ**：最新的感知量化，质量更好
- **ollama pull qwen2.5:1.5b** → 自动下载量化版

## 面试话术

> 不需要背公式，但可以这样说：

**"Flash Attention 我知道，它不是改模型结构，而是优化 Attention 的计算方式——利用 GPU 的 SRAM 分块计算，减少 HBM 的读写。速度提升 2-4 倍，精度完全不变，HuggingFace 里一行代码就能开了。实际项目里遇到长文本推理时很有用。"**

**"GQA 是 Llama 2 开始用的，把多个 Query Head 分到一组共享 Key-Value。在几乎不影响效果的前提下，大幅减少了 KV Cache 的内存占用，推理速度和吞吐都有提升。"**

**"MOE 的原理是训练很多个专家，每次只激活其中一部分。比如 Mixtral 8x7B，总参数量 46B，但每次只激活 12B，所以速度和 12B 模型差不多，但效果接近 46B。"**
